# 푸시 알림센터 개선기 (2)

- GPT 요약
    
    ## 1. 인터페이스 간단화
    
    기존에는 판매자 / 구매자에게 보낼 파라미터와 코드를 개발자가 직접 정의하는 구조였습니다.
    
    ### 🔎 문제점
    
    - 어드민 페이지에서 판매자에게만 보내고 싶어 판매자 메시지만 정의했더라도, 코드에 반영되어 있지 않으면 전송이 불가능했습니다.
    - 판매자/구매자에게 전달할 메시지가 코드와 DB에 모두 반영되어야 했기 때문에 개발팀과 사업팀이 공동 책임을 지는 구조였습니다.
    - 기획자는 어드민 페이지가 있음에도 불구하고, 푸시 메시지 추가/삭제를 개발자에게 요청해야 하는 구조가 되었습니다.
    
    결국 이런 고민이 생겼습니다.
    
    > "관리 책임을 어드민(DB) 중심으로 옮기고, 개발자는 책임을 줄일 수 있는 구조는 없을까?"
    > 
    
    또한 푸시 전송 모듈을 사용하는 개발자들로부터
    
    “보내는 인터페이스가 까다롭다”는 의견도 있었습니다.
    
    그래서 개발자 인터페이스는 단순화하고,
    
    나머지 제어 권한은 DB가 책임지는 구조로 방향을 잡았습니다.
    
    ---
    
    ## 기존 방식의 문제
    
    기존 코드는 메시지 타입별로 DTO와 분기 코드가 계속 추가되는 구조였습니다.
    
    ```java
    else if (queueData.getData() instanceof PushMessageRequest.PurchaseConfirmSeller
    	&& PushCodeEnum.MARKET.PURCHASE_CONFIRM.PURCHASE_CONFIRM_COMPLETED_SELLER.getCode() == queueData.getCode()) {
    
    	PushMessageRequest.PurchaseConfirmSeller body =
    		(PushMessageRequest.PurchaseConfirmSeller) queueData.getData();
    }
    ```
    
    메시지 하나가 추가되면:
    
    - ENUM 추가
    - DTO 추가
    - 리시버 분기 추가
    - 쿼리 ID 추가
    - 메시지 조합 로직 추가
    
    이 반복되었습니다.
    
    특히 구매자(BUYER) / 판매자(SELLER) 각각 정의되면
    
    리시버 코드는 계속 길어질 수밖에 없는 구조였습니다.
    
    ---
    
    ## 개선 방향
    
    복잡한 타입 정의 대신, 푸시 대상을 3가지로 단순화했습니다.
    
    - MARKET (기프티콘 장터)
    - TICKET (티켓 장터)
    - OTHER (단독 대상)
    
    그리고 각 대상 안에서 "이벤트" 기준으로 분리하기로 했습니다.
    
    ---
    
    ## 개선된 구조
    
    ```java
    pushQueueServiceV2.eventTrigger(
    	PushMessageRequestV2.MARKET.of(
    		PUSH_MGMT_MESSAGE_TYPE.MARKET.PRUCHASE,
    		16005
    	)
    );
    ```
    
    의미는 명확합니다.
    
    - MARKET 도메인에서
    - PURCHASE 이벤트가 발생했고
    - PK 16005 거래에 대한 푸시를 전송한다
    
    이제 개발자는 **이벤트만 트리거**하면 됩니다.
    
    ---
    
    ## PushMessageRequestV2 구조
    
    핵심은 공통 추상 클래스입니다.
    
    ```java
    public static abstract class Common {
    	private String category;
    	private CommonEnum.pushLocType locType;
    	private CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.PushMessageType pushMessageType;
    }
    ```
    
    이를 상속받는 구조는 다음과 같습니다.
    
    - TICKET
    - MARKET
    - OTHER
    
    그리고 확장을 위한 CUSTOM 구조:
    
    ```java
    public interface CUSTOM {
    	Object param(String name);
    }
    ```
    
    CUSTOM 타입은 Map 기반으로 추가 파라미터를 전달할 수 있도록 설계했습니다.
    
    단, 핵심 비즈니스 로직은 반드시
    
    MARKET / TICKET / OTHER 타입을 기준으로 동작하도록 유지했습니다.

    # 2. 기획자에게 권한을 줄 수 없을까?
    
    기존 구조에서는 메시지가 추가될 때마다 리시버 코드의 분기가 늘어났습니다.
    
    ```java
    else if (queueData.getData() instanceof PushMessageRequest.PurchaseConfirmSeller
    	&& PushCodeEnum.MARKET.PURCHASE_CONFIRM.PURCHASE_CONFIRM_COMPLETED_SELLER.getCode() == queueData.getCode()) {
    }
    ```
    
    메시지 타입 하나마다:
    
    - 타입 정의
    - DTO 정의
    - 쿼리 ID 정의
    - 템플릿 ID 정의
    - 리시버 분기 추가
    
    이 구조는 메시지가 늘어날수록 선형적으로 복잡해졌습니다.
    
    또한 DB 템플릿이 존재하더라도
    
    결국 코드가 반영되어야만 동작하는 구조였기 때문에
    
    기획팀이 자율적으로 운영하기 어려웠습니다.
    
    ---
    
    ## 개선된 리시버 구조
    
    ```java
    @RabbitListener(
    	queues = RabbitMQConstants.PALRAGO_PUSH_SEND_V2_QUEUE,
    	concurrency = "3"
    )
    private void send(GenericQueue<? extends PushMessageRequestV2.Common> genericQueue) {
    	PushMessageRequestV2.Common data = genericQueue.getData();
    
    	try {
    		pushAction(
    			data,
    			(userNo, pushMgmtDt, req, partnership) ->
    				send(userNo, pushMgmtDt, req, partnership)
    		);
    	} catch (Exception e) {
    		logger.error("푸시 전송에 실패하였습니다. {}", data);
    	}
    }
    ```
    
    공통 로직을 `pushAction()` 으로 모았습니다.
    
    ---
    
    ## 핵심 로직
    
    ```java
    private void pushAction(
    	PushMessageRequestV2.Common data,
    	QuadConsumer<Integer, PushMgmtDT, PushMessageRequestV2.Common, String> action
    ) {
    	PushReceiverMetaInfo receiver = getPushReceiverMetaInfo(data);
    
    	if (data instanceof PushMessageRequestV2.TICKET || data instanceof PushMessageRequestV2.MARKET) {
    		pushMgmtService
    			.selectUsableList(
    				data.getPushMessageType().getCode(),
    				receiver.buyUserPartnership(),
    				"BUYER"
    			)
    			.forEach(buyUserPushDT ->
    				action.accept(
    					receiver.buyUserNo(),
    					buyUserPushDT,
    					data,
    					receiver.buyUserPartnership()
    				)
    			);
    
    		pushMgmtService
    			.selectUsableList(
    				data.getPushMessageType().getCode(),
    				receiver.sellUserPartnership(),
    				"SELLER"
    			)
    			.forEach(sellUserPushDT ->
    				action.accept(
    					receiver.sellUserNo(),
    					sellUserPushDT,
    					data,
    					receiver.sellUserPartnership()
    				)
    			);
    	}
    
    	if (data instanceof PushMessageRequestV2.OTHER) {
    		pushMgmtService
    			.selectUsableList(
    				data.getPushMessageType().getCode(),
    				receiver.partnership(),
    				"OTHER"
    			)
    			.forEach(pushDT ->
    				action.accept(
    					receiver.userNo(),
    					pushDT,
    					data,
    					receiver.partnership()
    				)
    			);
    	}
    }
    ```
    
    이 구조의 특징은 다음과 같습니다.
    
    - 이벤트 타입 기준으로 템플릿 조회
    - DB에 정의된 템플릿을 그대로 반복 전송
    - 리시버는 더 이상 메시지별 분기를 가지지 않음
    
    이제 기존에 정의된 이벤트에 대해서는
    
    기획팀이 어드민에서 템플릿을 추가/비활성화하여 제어할 수 있습니다.
    
    ---
    
    # 3. 리시버 코드를 줄일 수 있었던 핵심 이유
    
    기존 리시버 코드가 길어진 원인은
    
    “메시지 타입마다 쿼리 ID가 달랐기 때문”이었습니다.
    
    예:
    
    ```
    push.requestRefundSeller
    push.requestPurchaseConfirmSeller
    ...
    ```
    
    문자열 기반 쿼리 ID가 타입마다 **흩어져 있었고**,
    
    이를 추상화하기 어려웠습니다.
    
    ---
    
    ## 해결 방법
    
    이벤트 타입 ENUM 내부에 쿼리 ID를 포함시켰습니다.
    
    ```
    PURCHASE("구매",1,null,"push.v2.purchaseSeller")
    ```
    
    그리고 호출은 단순합니다.
    
    ```
    pushQueueServiceV2.eventTrigger(
    	PushMessageRequestV2.MARKET.of(
    		PUSH_MGMT_MESSAGE_TYPE.MARKET.PRUCHASE,
    		16005
    	)
    );
    ```
    
    이제:
    
    - 이벤트 정의
    - 코드
    - 쿼리 ID
    
    가 하나의 ENUM에 응집되었습니다.
    
    그 결과:
    
    - 리시버는 공통 모듈이 되었고
    - 더 이상 수정되지 않는 핵심 영역이 되었으며
    - 책임은 명확히 분리되었습니다
    
    ---
    
    # 정리
    
    이번 개선의 핵심은 단순히 코드 라인을 줄인 것이 아니라:
    
    - 개발자는 이벤트만 트리거
    - 기획팀은 DB 템플릿을 관리
    - 리시버는 공통 처리만 수행
    
    하도록 책임을 재정렬한 것입니다.
    
    결과적으로:
    
    - 인터페이스는 단순해졌고
    - 리시버는 안정되었으며
    - 기획팀의 자율성은 높아졌습니다.

## 1. 인터페이스 간단화

기존에는 판매자.구매자에게 보낼 파라미터와 코드를 개발자가 직접 정의했어요.

**문제점**

- 이러다보니 어드민 페이지에서 어떤 경우에는 판매자한테만 보내고 싶어서 판매자에 대한 메세지만 정의해놨지만, 코드 반영이 되어있지 않으면 보내기가 불가능했어요.
- 또한, 판.구매자에게 전달할 메세지가 코드와 DB에 둘다 반영이 되어야해서 코드, DB(사업팀)가 책임을 공동으로 지다보니 기획자는 푸시 메세지에 관한 내용의 추가나 삭제를 어드민 페이지가 있음에도 불구하고 개발자에게 모두 위임하는 구조가 되었어요.

그럼 어떤식으로 관리에 대한 책임을 어드민에만 주고 개발자는 책임을 덜어내는 방식이 뭐지? 라는 고민이 들기 시작했어요.

이 문제와 복합적으로 푸시 전송 모듈을 사용하는 개발자들이 보내기가 까다롭다는 의견을 반영하여 개발자들에게는 인터페이스를 간단화 시키고 나머지 책임은 DB에서 책임지는 것으로 구상을 한 결과 개발자들이 사용하는 모듈의 인터페이스의 간단화를 추진했어요.

**기존 코드**처럼 복잡하게 클래스 타입을 만들어서, 각각의 파라미터를 넣기 보다는 판.구매자가 구분되는 "기프티콘 장터"면 MARKET, "티켓 장터"면 TICKET, 각 개별로 받아야하는 푸시에 대한것이라면 OTHER 클래스로 정의했어요.

```java
package kr.co.kpcard.palrago.common.push.domain;

import java.util.Map;

import kr.co.kpcard.palrago.common.constants.CommonCodeMap;
import kr.co.kpcard.palrago.common.domain.CommonEnum;
import lombok.AccessLevel;
import lombok.AllArgsConstructor;
import lombok.Getter;
import lombok.NoArgsConstructor;
import lombok.ToString;

public class PushMessageRequestV2 {
	@Getter
	@ToString
	@NoArgsConstructor(access = AccessLevel.PROTECTED)
	@AllArgsConstructor
	public static abstract class Common {
		private String category;
		private CommonEnum.pushLocType locType;        // 푸시 보내는 위치
		private CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.PushMessageType pushMessageType; // 푸시 타입 코드

		protected Common(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.PushMessageType pushMessageType, String category) {
			this.pushMessageType = pushMessageType;
			this.category = category;
		}

		protected void autoSetLocType(CommonEnum.pushLocType resolvePushLocType) {
			this.locType = resolvePushLocType;
		}

		protected void autoSetPushMessageType(
			CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.PushMessageType resolvePushMessageType) {
			this.pushMessageType = resolvePushMessageType;
		}
	}

	/**
	 * 티켓 공통 REQUEST DTO
	 */
	@Getter
	@ToString
	@NoArgsConstructor(access = AccessLevel.PROTECTED)
	public static class TICKET extends Common {
		private int ticketSubSeqNo;

		public TICKET(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.TICKET pushMessageType, int ticketSubSeqNo) {
			super(pushMessageType, "TICKET");
			this.ticketSubSeqNo = ticketSubSeqNo;
		}

		public static TICKET of(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.TICKET pushMessageType, int ticketSubSeqNo) {
			return new TICKET(pushMessageType, ticketSubSeqNo);
		}
	}

	/**
	 * 장터 공통 REQUEST DTO
	 */

	@Getter
	@ToString
	@NoArgsConstructor(access = AccessLevel.PROTECTED)
	public static class MARKET extends Common {
		private int sellSubSeqNo;

		protected MARKET(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.MARKET pushMessageType, int sellSubSeqNo) {
			super(pushMessageType, "MARKET");
			this.sellSubSeqNo = sellSubSeqNo;
		}

		public static MARKET of(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.MARKET pushMessageType, int sellSubSeqNo) {
			return new MARKET(pushMessageType, sellSubSeqNo);
		}

	}

	/**
	 * 단독 공통 REQUEST DTO
	 */
	@Getter
	@ToString
	@NoArgsConstructor(access = AccessLevel.PROTECTED)
	public static class OTHER extends Common {
		private int userNo;

		public OTHER(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.OTHER pushMessageType, int userNo) {
			super(pushMessageType, "OTHER");
			this.userNo = userNo;
		}

		public static OTHER of(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.OTHER pushMessageType, int userNo) {
			return new OTHER(pushMessageType, userNo);
		}

	}

```

또한 추가적으로 아래와 같이 고정적인 필요한 값 외에 상황에서 다양한 파라미터를 받아야한다면 CUSTOM 클래스를 만들어서 파라미터를 사용자 입맛에 맞게 전달할 수 있도록 타입을 추가했어요.

```java
public interface CUSTOM {
	public Object param(String name);
}

@Getter
@ToString
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public static class MARKET_CUSTOM extends MARKET implements CUSTOM {

	private Map<String, Object> customParameter;

	protected MARKET_CUSTOM(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.MARKET pushMessageType, int sellSubSeqNo, Map<String, Object> customParameter) {
		super(pushMessageType, sellSubSeqNo);
		this.customParameter = customParameter;
	}

	public static MARKET_CUSTOM of(CommonCodeMap.PUSH_MGMT_MESSAGE_TYPE.MARKET pushMessageType, int sellSubSeqNo, Map<String, Object> customParameter) {
		return new MARKET_CUSTOM(pushMessageType, sellSubSeqNo, customParameter);
	}

	@Override
	public Object param(String name) {
		return customParameter.get(name);
	}
}

```

그럼 해당 클래스들은 현저히 줄게되고 MARKET, TICKET, OTHER 이 푸시 대상을 찾는 기준이 될 것이고 그 안에 공통 파라미터로 "이벤트" 에 대한 기준을 나누기로 했어요.

즉, 아래와 코드가 같아져요.

```java
pushQueueServiceV2.eventTrigger(
	PushMessageRequestV2.MARKET.of(
		PUSH_MGMT_MESSAGE_TYPE.MARKET.PRUCHASE,
		16005
	)
);
```

`PUSH_MGMT_MESSAGE_TYPE.MARKET.PRUCHASE` 은 어떤 이벤트에 대한 트리거라는 뜻이에요. 즉, 장터에서 구매가 일어났어!

어떤 구매에 대한 이벤트냐면 `16005` PK 를 가진 거래에 대한 푸시를 보내야해! 라는 뜻이에요.

## 2. 기획자에게 권한을 좀 주시면 안될까요? 불편해요.

기존 코드는 타입별로 너무나 많은 코드를 정의하고 있어 if - else 가 보낼 푸시 메세지가 늘어날수록 많아져야하는 구조로 되어있었어요.

```java
else if (queueData.getData() instanceof PushMessageRequest.PurchaseConfirmSeller
			&& PushCodeEnum.MARKET.PURCHASE_CONFIRM.PURCHASE_CONFIRM_COMPLETED_SELLER.getCode()
			== queueData.getCode()) {

			// 캐스팅 요청 파라미터
			PushMessageRequest.PurchaseConfirmSeller body = (PushMessageRequest.PurchaseConfirmSeller)queueData.getData();
```

`PURCHASE_CONFIRM.PURCHASE_CONFIRM_COMPLETED_SELLER` 해당 타입은 구매 확정당시에 판매자에게 보낼 enum 이자 그 안에 필요한 파라미터를 정해 메세지당, 타입이 하나가 정의되며, 파라미터 또한 그 타입에 맞는 클래스가 생성이 되어야 했어요.

해당 코드만 보시더라도 그러면? 구매자 (`BUYER`) 에게 보낼 푸시도 정의가 되면 리시버(컨슈머) 코드도 길어지고 그에 맞는 파라미터에 핏하게 맞는 클래스도 정의되어야하며,,,, 생각만해도 엄청많은 코드가 푸시메세지 한줄 때문에 들어가야하지 않을까요? 라는 생각을 하게 됩니다.
또, 이렇게 생각하면 DB는 큰 의미없이 개발자에게 컨펌을 받아야하는 기획팀의 불편함도 있을 수 밖에 없을 것 같아요.

그래서 개선된 코드는 아래와 같아요.

```java
@RabbitListener(
	queues = RabbitMQConstants.PALRAGO_PUSH_SEND_V2_QUEUE,
	concurrency = "3"
)
private void send(GenericQueue<? extends PushMessageRequestV2.Common> genericQueue) {
	PushMessageRequestV2.Common data = genericQueue.getData();

	try {
		pushAction(
			data,
			(userNo, pushMgmtDt, req, partnership) ->
				send(userNo, pushMgmtDt, req, partnership)
		);
	} catch (Exception e) {
		logger.error("푸시 전송에 실패하였습니다. {}", data);
	}
}

@RabbitListener(queues = RabbitMQConstants.PALRAGO_PUSH_SAVE_V2_QUEUE)
private void save(GenericQueue<? extends PushMessageRequestV2.Common> genericQueue) {
	PushMessageRequestV2.Common data = genericQueue.getData();

	try {
		pushAction(
			data,
			(userNo, pushMgmtDt, req, partnership) ->
				save(userNo, pushMgmtDt, req, partnership)
		);
	} catch (Exception e) {
		logger.error("푸시 저장에 실패하였습니다. {}", data);
	}
}
```

현저하게 로직이 줄어들었고 공통로직을 함수형 인터페이스를 통해서 하나로 묶어 개선하였습니다.

여기서 핵심은 이부분인데요.

```java
private void pushAction(
	PushMessageRequestV2.Common data,
	QuadConsumer<Integer, PushMgmtDT, PushMessageRequestV2.Common, String> action
) {
	// 1. 수신 대상 정보(유저번호, 파트너사) 추출
	PushReceiverMetaInfo receiver = getPushReceiverMetaInfo(data);

	// 2. 장터 거래(티켓/기프티콘) 대상 처리
	if (data instanceof PushMessageRequestV2.TICKET || data instanceof PushMessageRequestV2.MARKET) {
		// BUYER
		pushMgmtService
			.selectUsableList(
				data.getPushMessageType().getCode(),
				receiver.buyUserPartnership(),
				"BUYER"
			)
			.forEach(
				buyUserPushDT -> action.accept(
					receiver.buyUserNo(),
					buyUserPushDT,
					data,
					receiver.buyUserPartnership()
				)
			);

		// SELLER
		pushMgmtService
			.selectUsableList(
				data.getPushMessageType().getCode(),
				receiver.sellUserPartnership(),
				"SELLER"
			)
			.forEach(
				sellUserPushDT -> action.accept(
					receiver.sellUserNo(),
					sellUserPushDT,
					data,
					receiver.sellUserPartnership()
				)
			);
	}

	// 3. 단독 대상 처리
	if (data instanceof PushMessageRequestV2.OTHER) {
		pushMgmtService
			.selectUsableList(
				data.getPushMessageType().getCode(),
				receiver.partnership(),
				"OTHER"
			)
			.forEach(
				pushDT -> action.accept(
					receiver.userNo(),
					pushDT,
					data,
					receiver.partnership()
				)
			);
	}
}
```

푸시에 관한 액션을 진행할 때 무작정 메세지 템플릿을 가져와서 그냥 일단 forEach로 나오는 템플릿대로 다 전송합니다.
이는 DB (즉, 기획자) 에게 책임을 다 전가할 수 있는 방식이에요. 기획팀에서 어느 경우에 추가적으로 보내고 싶다는 것이 결정되면 그건 개발자가 추가적으로 개발을 해야하겠지만(코드 추가적인 면), 기존에 미리 정의되어있는 상황에 대해서는 정말 마음대로 어드민이 직접 정의하여 보낼수도 있어요.

## 3. 리시버 코드를 파격적으로 없애자.

기존 리시버 코드가 파격적으로 길어진데에 있어서는 "파라미터" 가존재하는 메세지 템플릿들 때문이였어요.
예를들어, 아래와 같아요.

```java
} else if (queueData.getData() instanceof PushMessageRequest.PurchaseConfirmSeller
			&& PushCodeEnum.MARKET.PURCHASE_CONFIRM.PURCHASE_CONFIRM_COMPLETED_SELLER.getCode()
			== queueData.getCode()) {

			// 캐스팅 요청 파라미터
			PushMessageRequest.PurchaseConfirmSeller body = (PushMessageRequest.PurchaseConfirmSeller)queueData.getData();

			// 파트너십 조회 후 그에 따른 푸시 메세지 템플릿 메타 정보 추출
			Optional.ofNullable(selectPushMetaInfoMarket(queueData, body.getSellSubSeqNo()))
				.ifPresentOrElse(
					pushMetaInfo -> {

					body.getSellSubSeqNo(),

					.
					.	(생략)
					send(message)
	} else if (queueData.getData() instanceof PushMessageRequest.RequestRefundSeller

					.

						? "push.requestRefundSeller" : "push.requestRefundSellerAsp",

					. (생략)
					.

```

`push.requestRefundSeller` 원인은 쿼리 때문이였어요. 이 쿼리 ID 가 어디에 생성이 되어야 코드가 줄 수 있는지를 고려하지 않았을 뿐더러 위에서 앞서 말했던 것 처럼 메세지 템플릿에 타입이 하나하나 핏하게 맞춰지다보니, 그에 맞는 쿼리ID 도 필요했어요. 코드를 뭔가 마술처럼 잘 짠다고 해도, 문자열을 추상화시킨다는건 어려웠어요.

그래서 저희는 이벤트 타입에 EX). `PUSH_MGMT_MESSAGE_TYPE.MARKET.PRUCHASE` ENUM 을 만들어서 내부에 공통으로 쿼리 ID 를 선언했어요.

```
pushQueueServiceV2.eventTrigger(
			PushMessageRequestV2.MARKET.of
				(PUSH_MGMT_MESSAGE_TYPE.MARKET.PRUCHASE, 16005));
```

해당 ENUM 에는 `PURCHASE("구매", 1, null, "push.v2.purchaseSeller"),` 의 형태로 타입이 선언되어지고 필드를 하나 만들어서 쿼리 ID 를 타입에 직접 정의하고 해당 타입에서 쿼리 ID 를 꺼내어 마이바티스 쿼리를 하는 식으로 구성하니, 타입에 적당한 데이터들이 몰리고 리시버는 공통화를 잘하여 절대 수정되지 않을 핵심 모듈로서의 역할을 하게 됐어요.

즉, 이 과정에서 DB는 기획자에게 책임을 교통정리하였고, 개발자가 가져야할 책임은 ENUM (타입) 에 몰아준 것이죠. 이렇게 되니 한결 더 수월하고 깔끔하였습니다.
