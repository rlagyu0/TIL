# DDD-같이읽기 애그리거트

### 애그리거트

관계가 있는 것들에 대한 집합.

### 애그리거트의 기준

애그리거트에 속한 객체는 보통 생며주기가 연관 객체끼리 같은 생명주기를 가짐.

예를 들어 주문이라는 애그리거트가 존재할 때, 배송 정보, 배송받는 사람에 대한 정보가 있다. 그런데 배송 정보가 존재하면 배송 받느 사람에 대한 정보도 당연히 존재할 것이다.

반면에 상품 ↔ 리뷰 와의 관계에서는 같은 애그리거트라고 의심할 수 있다.

- 상품 상세를 보여줄 때 리뷰도 같이 볼 수 있게 해야한다.
- 하지만? 상품이 있다고 해서 누군가는 리뷰를 꼭 적어야만 하는 것은 아니다.

위의 정보에 따라 상품과 리뷰는 애그리거트가 아니다.

### 애그리거트의 일관성

애그리거트는 같은 성격의 도메인의 집합인 만큼 각각의 객체가 데이터 일관성을 이뤄야함.

- setter 금지
- setter 금지의 연장선으로 각 객체들은 불변해야한다.
- 응용 로직에 도메인 로직을 나열하지말고, 루트 애그리거트에 도메인 로직의 책임을 묻자.
    - 즉, 루트 애그리거트에 도메인 규칙을 정의하고 따르자.

### 애그리거트 트랜잭션

한 트랜잭션에서 애그리거트는 하나만 수정되어져야한다.

- 두개 이상의 애그리거트가 수정될 경우에는 (처리할 수 있는 도메인의 갯수가 늘어날 수록 DB LOCK) 성능에 이상이 생길 수 있다.
- **결합도가 높아져서 수정 비용이 증가.**
    
    ex → 
    
    ```java
    class Order {
    	private Orderer orderer;
    	private ShippingInfo shippingInfo;
    	
    	public void 배송지_정보_변경(newShippingInfo, boolean 기본_배송지도_변경할건지?) {
    		주문에 대한 배송지 변경 로직 start
    		.
    		.
    		주문에 대한 배송지 변경 로직 end
    		if(기본_배송지도_변경할건지?) {
    			 orderer.getMember().changeAddress(newShippingInfo ...)
    		}
    	}
    }
    ```
    

**굳이 한 트랜잭션에서 두개이상의 애그리거트를 변경해야한다면 응용 계층에서 분리하여 수정하는 것을 추천**

### 애그리거트 참조

```java
Order -> Orderer -> **Member**
```

해당 구조처럼 주문이라는 애그리거트와 멤버라는 애그리거트의 경계에서 참조 구조일 때 참조의 오남용은 결합도와 성능면에서 불리할 수 있음.

- 성능면에서 Eager/Lazy 로딩 같은 것들을 고려해야함.
- 하위 도메인이 각자 다른 DBMS 를 참조할 경우 걸림돌이 될 수 있음.
- 다른 애그리거트에 접근하기 쉽다는 것은 수정도 용이하다는 것이다. 결합도가 증가하여 변경에 어려움이 있다.

이에 대한 해결 방안으로 id 값을 참조할 수 있도록 한다.

```java
class Orderer {
	private	MemberId  memberId;
}
```

**장점**

- 기존 애그리거트를 참조하면 Order 를 조회하는 순간 member 도 가져왔어야 했지만. 가져올 필요가 없어진다. Lazy Loading 강제화
- 위의 이유에 따라 memberId 로 내가 조회하고 싶을 때 조회할 수 있는 로직을 호출하면 됨.

**단점**

- 하지만 위의 장점도 있지만 장점으로 인한 단점도 존재한다.
- 만약 1:N 구조일 경우 N+1 문제 발생할 수 있음. (Join 으로 해결)

### 애그리거트 팩토리

애그리거트는 객체를 생성하는 것도 가능함.

ex. 상점이 상품을 개제할 때 로직

→

```java
public Long registerProduct(storeRepository, productRepository) {
	store =	storeRepository.findId(상점 id);
	if(store.isBlock()) {
		throw Exception
	}
	product =	new Product();
	productRepositoryt.save(product);
	return product.id();
}
```

이런 것도 이상할 것 없이 좋지만, store가 도메인 로직에 일부 가지고 들어가는 것이 가능.

```java
class store {
	public Product createProduct(...) {
	
		if(store.isBlock()) {
			throw Exception
		}
		return new Product();
	}
}
```

**추가적으로 많은 정보가 더필요할 경우에는 Factory 클래스를 만들어서 사용할 수 있음.**
